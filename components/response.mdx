# Response Component

The Response component provides a structured way to document HTTP API responses, including status codes, headers, and response body examples.

## Overview

Use Response components to document API endpoint responses, helping developers understand what to expect when calling your API endpoints.

## Properties

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `status` | `number` | Yes | HTTP status code (200, 404, 500, etc.) |
| `statusText` | `string` | No | Status text description |
| `headers` | `object` | No | Response headers object |
| `children` | `ReactNode` | No | Response body or additional content |
| `title` | `string` | No | Custom title for the response |
| `className` | `string` | No | Additional CSS classes for styling |

## Usage Examples

### Success Response

```mdx
<Response status={200} title="Successful User Creation">
  <CodeBlock language="json">
  {`{
    "id": "user_123",
    "name": "John Doe",
    "email": "john@example.com",
    "role": "user",
    "createdAt": "2024-01-15T10:30:00Z",
    "updatedAt": "2024-01-15T10:30:00Z"
  }`}
  </CodeBlock>
</Response>
```

### Response with Headers

```mdx
<Response 
  status={201} 
  statusText="Created"
  headers={{
    "Content-Type": "application/json",
    "Location": "/api/users/user_123",
    "X-RateLimit-Remaining": "99"
  }}
>
  <CodeBlock language="json">
  {`{
    "id": "user_123",
    "name": "Jane Doe",
    "email": "jane@example.com",
    "status": "active"
  }`}
  </CodeBlock>
</Response>
```

### Error Response

```mdx
<Response status={400} title="Validation Error">
  <CodeBlock language="json">
  {`{
    "error": {
      "code": "VALIDATION_ERROR",
      "message": "Invalid input data",
      "details": [
        {
          "field": "email",
          "message": "Email format is invalid"
        },
        {
          "field": "password",
          "message": "Password must be at least 8 characters"
        }
      ]
    }
  }`}
  </CodeBlock>
</Response>
```

### Response with Field Documentation

```mdx
<Response status={200} title="User Profile Response">
  <CodeBlock language="json">
  {`{
    "id": "user_123",
    "profile": {
      "name": "John Doe",
      "email": "john@example.com",
      "avatar": "https://example.com/avatars/user_123.jpg",
      "preferences": {
        "theme": "dark",
        "notifications": true,
        "language": "en"
      }
    },
    "subscription": {
      "plan": "pro",
      "status": "active",
      "expiresAt": "2024-12-31T23:59:59Z"
    }
  }`}
  </CodeBlock>
  
  
  <ResponseField path="id" type="string">
    Unique identifier for the user.
  </ResponseField>
  
  <ResponseField path="profile.name" type="string">
    The user's display name.
  </ResponseField>
  
  <ResponseField path="profile.email" type="string">
    The user's email address.
  </ResponseField>
  
  <ResponseField path="subscription.plan" type="string">
    Current subscription plan: <code>free</code>, <code>pro</code>, or <code>enterprise</code>.
  </ResponseField>
</Response>
```

### Empty Response

```mdx

  </Paragraph>
</Response>
```

## Common Status Codes

### 2xx Success
- **200 OK** - Request successful, response body included
- **201 Created** - Resource created successfully
- **202 Accepted** - Request accepted for processing
- **204 No Content** - Request successful, no response body

### 4xx Client Errors
- **400 Bad Request** - Invalid request data
- **401 Unauthorized** - Authentication required
- **403 Forbidden** - Access denied
- **404 Not Found** - Resource not found
- **422 Unprocessable Entity** - Validation errors

### 5xx Server Errors
- **500 Internal Server Error** - Server encountered an error
- **502 Bad Gateway** - Invalid response from upstream server
- **503 Service Unavailable** - Server temporarily unavailable

## Accessibility

- Uses semantic markup for response information
- Provides clear status code and description
- Supports keyboard navigation and focus management
- Compatible with screen readers

## Best Practices

- Always include the correct status code
- Provide realistic response examples
- Document all possible response scenarios
- Include error response examples
- Use consistent response structure across endpoints
- Document response headers when relevant
