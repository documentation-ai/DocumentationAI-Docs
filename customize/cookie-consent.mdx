description: >-
  Set up cookie consent to control when integrations, custom scripts, and
  telemetry load on your documentation site. Works with any consent management
  platform (CMP).
---
Control which integrations, custom scripts, and Documentation.ai telemetry require user consent before loading. Without consent categories, everything loads normally. Once you add `integrations.cookies`, Documentation.ai uses those categories to decide what loads immediately and what waits for consent.

This works with any consent management platform (CMP) — CookieBot, OneTrust, CookieYes, Osano, Termly, or a custom consent banner.

## How it works

1. You add your CMP banner as a [custom script](/customize/custom-scripts) so visitors can make a consent choice.

2. You define consent categories under `integrations.cookies` using the cookie or localStorage key-value pairs your CMP writes.

3. You optionally assign those categories to integrations, custom scripts, or telemetry. If you do not assign a category, Documentation.ai falls back to the built-in default for that item.

4. After a visitor grants a category, only the integrations, scripts, and telemetry assigned to that category can load.

If you do not configure `integrations.cookies`, Documentation.ai does not gate loading and your existing integrations and scripts keep their normal behavior.

<Callout kind="info" collapsed="false">
  Documentation.ai does not include a built-in consent banner. Add your CMP's script via [Custom Scripts](/customize/custom-scripts). The cookie consent configuration here only controls when other integrations, scripts, and telemetry are allowed to run.
</Callout>

## Choose your setup

Not every site needs the full cookie consent configuration. Choose the approach that fits your situation.

### I use a CMP with auto-blocking

Some CMPs automatically block third-party scripts at the network level. In that setup, you usually do not need to configure `integrations.cookies` for those providers.

**What to do:** Add your CMP banner as a [custom script](/customize/custom-scripts). Leave cookie consent categories off unless you also want Documentation.ai to gate built-in telemetry.

The main exception is **Documentation.ai telemetry**. If you want telemetry to wait for consent or stay off entirely, configure it in [Telemetry](#telemetry).

### I use a CMP without auto-blocking

Some CMPs show a banner and store the visitor's choice, but they do not block scripts on their own. In that case, use Documentation.ai's cookie consent settings to gate what loads.

**What to do:** Add your CMP script, define consent categories, then assign categories to the integrations and scripts that should wait for consent.

### I only want to control Documentation.ai telemetry

If you do not need consent gating for third-party integrations, you can still disable telemetry or move it to a consent category.

**What to do:** Go to **Site Config > Integrations > Telemetry** in the Editor, or add telemetry settings directly to `documentation.json`.

## Prerequisites

- A documentation project with [integrations](/integrations/overview) or [custom scripts](/customize/custom-scripts) configured
- A CMP account (CookieBot, OneTrust, CookieYes, Osano, Termly, and similar tools) or a custom consent banner script

## Step 1: Add your CMP banner script

Your CMP provides a JavaScript snippet or script URL to show the consent banner. Add it as a custom script so it loads on every page.

### From the Editor

<Steps>
  <Step title="Get the embed code from your CMP" icon="clipboard" title-type="p">

    Log in to your CMP dashboard and copy the JavaScript embed snippet. It typically looks like this:

    ```js
    var script = document.createElement("script");
    script.src = "https://consent.cookiebot.com/uc.js?cbid=example-consent-id";
    document.head.appendChild(script);
    ```

  </Step>

  <Step title="Create a script file" icon="file-code" title-type="p">

    In the Editor, go to **Site Config > Custom Scripts**. Click **Add Script > Create New Script**. Name it something like `cmp-banner.js` and paste your CMP's embed code into the editor.

  </Step>

  <Step title="Publish" icon="upload" title-type="p">

    Click **Publish**. The consent banner now appears on every page of your documentation site.

  </Step>
</Steps>

### Via documentation.json

If your CMP provides a direct script URL, reference it in `documentation.json`:

```json
{
  "scripts": [
    { "src": "https://consent.cookiebot.com/uc.js?cbid=example-consent-id" }
  ]
}
```

Or if you saved the embed code as a local `.js` file in your project:

```json
{
  "scripts": [
    { "src": "scripts/cmp-banner.js" }
  ]
}
```

<Callout kind="info" collapsed="false">
  Scripts without a `category` field default to `essential`, so they always load. Keep your CMP banner script essential. If you assign it a consent category, the banner can be blocked before visitors get a chance to choose.
</Callout>

## Step 2: Configure consent categories

Each consent category maps to the cookie or localStorage value your CMP writes after a visitor grants consent.

### Where do the key and value come from?

You have two options.

**Option A: Use your CMP's built-in values.** Every CMP stores consent in a known cookie or localStorage entry. See the [CMP reference](#cmp-reference) below for common examples.

**Option B: Write your own values.** If your CMP supports callbacks, you can store a simple value such as `analytics_consent=granted` and match against that instead.

### From the Editor

<Steps>
  <Step title="Open Site Config" icon="settings" title-type="p">

    In the Editor, click **Site Config** in the top toolbar.

  </Step>

  <Step title="Navigate to Cookies" icon="cookie" title-type="p">

    Select **Cookies** from the left sidebar.

  </Step>

  <Step title="Enable and add categories" icon="toggle-right" title-type="p">

    Turn on cookie consent. An `analytics` category is created by default. Add more categories such as `marketing` or `functional`, or create your own.

  </Step>

  <Step title="Fill in key and value" icon="key" title-type="p">

    For each category, enter:

    - **Key** — the cookie name or localStorage key your CMP writes, such as `CookieConsent`
    - **Value** — the substring that signals consent for that category, such as `statistics:true`

  </Step>

  <Step title="Publish" icon="upload" title-type="p">

    Click **Publish** to deploy the consent configuration.

  </Step>
</Steps>

### Via documentation.json

```json
{
  "integrations": {
    "cookies": {
      "analytics": {
        "key": "CookieConsent",
        "value": "statistics:true"
      },
      "marketing": {
        "key": "CookieConsent",
        "value": "marketing:true"
      }
    },
    "ga4": {
      "measurementId": "G-ABC123DEF4",
      "category": "analytics"
    },
    "intercom": {
      "appId": "abc123xyz"
    }
  },
  "scripts": [
    { "src": "https://consent.cookiebot.com/uc.js?cbid=example-consent-id" }
  ]
}
```

<ParamField path="integrations.cookies" param-type="object" required="false" deprecated="false">

Each key is a consent category name you define. The value tells Documentation.ai what to look for in the browser.

  <Expandable title="Category properties" default-open="true">

    <ParamField path="key" param-type="string" required="true" deprecated="false">
      The cookie name or localStorage key that your consent banner writes to.
    </ParamField>

    <ParamField path="value" param-type="string" required="true" deprecated="false">
      The substring to look for in the stored value. Documentation.ai treats the category as granted when the stored value contains this string.
    </ParamField>

  </Expandable>

</ParamField>

## Step 3: Assign categories to integrations and scripts

### Integration categories

Each integration can include an optional `category` field. When you configure `integrations.cookies`, Documentation.ai applies these defaults unless you override them.

| Integration family                                                                                                      | Default category           |
| ----------------------------------------------------------------------------------------------------------------------- | -------------------------- |
| ga4, gtm, posthog, mixpanel, amplitude, segment, plausible, fathom, pirsch, heap, hightouch, hotjar, clarity, logrocket | `analytics`                |
| clearbit                                                                                                                | `marketing`                |
| intercom, crisp                                                                                                         | `essential` (always loads) |

If you do not configure `integrations.cookies`, these defaults do not block anything and integrations load normally.

To override a default, add a `category` field:

```json
"ga4": { "measurementId": "G-ABC123DEF4", "category": "marketing" }
```

You can also set categories from the Editor. When cookies are configured, each integration card shows a **Consent Category** option.

### Custom script categories

Custom scripts default to `essential` and always load. To gate a script behind consent, add a `category` field:

```json
{
  "scripts": [
    { "src": "https://consent.cookiebot.com/uc.js?cbid=example-consent-id" },
    { "src": "scripts/fonts.js" },
    { "src": "scripts/tracking.js", "category": "analytics" }
  ]
}
```

| Script            | Category              | Behavior                                                            |
| ----------------- | --------------------- | ------------------------------------------------------------------- |
| CMP banner script | `essential` (default) | Always loads so visitors can make a consent choice                  |
| `fonts.js`        | `essential` (default) | Always loads because no `category` is set                           |
| `tracking.js`     | `analytics`           | Only loads after analytics consent is granted                       |

<Callout kind="danger" collapsed="false">
  Never assign a consent category to your CMP banner script. If the banner itself waits for consent, visitors never see it and cannot grant consent.
</Callout>

## CMP reference

Below are example key-value pairs for popular CMPs. Copy the pattern that matches your provider into `integrations.cookies`.

### CookieBot

CookieBot stores consent in a cookie called `CookieConsent`.

```json
"cookies": {
  "analytics": { "key": "CookieConsent", "value": "statistics:true" },
  "marketing": { "key": "CookieConsent", "value": "marketing:true" },
  "functional": { "key": "CookieConsent", "value": "preferences:true" }
}
```

### OneTrust

OneTrust uses group IDs in a cookie called `OptanonConsent`.

```json
"cookies": {
  "analytics": { "key": "OptanonConsent", "value": "C0002:1" },
  "functional": { "key": "OptanonConsent", "value": "C0003:1" },
  "marketing": { "key": "OptanonConsent", "value": "C0004:1" }
}
```

### CookieYes

CookieYes stores consent in a cookie called `cookieyes-consent`.

```json
"cookies": {
  "analytics": { "key": "cookieyes-consent", "value": "analytics:yes" },
  "functional": { "key": "cookieyes-consent", "value": "functional:yes" },
  "marketing": { "key": "cookieyes-consent", "value": "advertisement:yes" }
}
```

### Osano

```json
"cookies": {
  "analytics": { "key": "osano_consentmanager", "value": "ANALYTICS\":\"ACCEPT" },
  "marketing": { "key": "osano_consentmanager", "value": "MARKETING\":\"ACCEPT" }
}
```

### Termly

```json
"cookies": {
  "analytics": { "key": "termly_consent_preferences", "value": "\"analytics\":true" },
  "marketing": { "key": "termly_consent_preferences", "value": "\"advertising\":true" }
}
```

### Custom consent banner

If you built your own consent banner that sets a localStorage value when the user consents:

```json
"cookies": {
  "analytics": { "key": "my_site_consent", "value": "accepted" }
}
```

<Callout kind="info" collapsed="false">
  Category names are flexible. Use names that match your CMP and your internal privacy model, such as `analytics`, `statistics`, `marketing`, `advertising`, `functional`, or `personalization`.
</Callout>

### Not sure which values to use?

If your CMP is not listed above, inspect the values directly:

1. Open your documentation site in a browser
2. Accept cookies in the consent banner
3. Open **Developer Tools** and inspect **Cookies** or **Local Storage**
4. Find the value your CMP wrote
5. Copy the substring that signals consent for each category

You do not need to match the full stored value. For example, if the cookie contains ``{statistics:true,marketing:false}``, using `statistics:true` is enough.

## Telemetry

Documentation.ai includes built-in page view tracking that powers your [analytics dashboard](/analytics/overview). Telemetry is enabled by default, but you control whether it runs immediately, waits for consent, or stays off entirely.

- If you do not configure `integrations.cookies`, telemetry starts tracking page views immediately.
- If you do configure `integrations.cookies`, telemetry waits for the category in `integrations.telemetry.category`.
- If you do not set `integrations.telemetry.category`, Documentation.ai uses `analytics` by default.

### Disable telemetry

```json
{
  "integrations": {
    "telemetry": { "enabled": "false" }
  }
}
```

### Assign telemetry to a different consent category

```json
{
  "integrations": {
    "telemetry": { "category": "statistics" },
    "cookies": {
      "statistics": { "key": "CookieConsent", "value": "statistics:true" }
    }
  }
}
```

<Callout kind="info" collapsed="false">
  Telemetry follows its own consent category. You can keep third-party analytics on one category, move Documentation.ai telemetry to another, or disable telemetry entirely.
</Callout>

You can also manage telemetry from the Editor under **Integrations > Telemetry**.

## Complete example

Here is a full configuration with CookieBot as the CMP, Google Analytics and PostHog using the default analytics category, Clearbit using the default marketing category, Intercom staying essential, and Documentation.ai telemetry explicitly tied to analytics consent.

```json
{
  "integrations": {
    "cookies": {
      "analytics": { "key": "CookieConsent", "value": "statistics:true" },
      "marketing": { "key": "CookieConsent", "value": "marketing:true" }
    },
    "ga4": { "measurementId": "G-ABC123DEF4" },
    "posthog": { "apiKey": "phc_example_9xmk7q2r" },
    "clearbit": { "publicKey": "pk_example_abc123" },
    "intercom": { "appId": "abc123xyz" },
    "telemetry": { "category": "analytics" }
  },
  "scripts": [
    { "src": "https://consent.cookiebot.com/uc.js?cbid=example-consent-id" },
    { "src": "scripts/fonts.js" },
    { "src": "scripts/ad-pixel.js", "category": "marketing" }
  ]
}
```

**What happens:**

| Script / Integration       | Category               | Behavior                      |
| -------------------------- | ---------------------- | ----------------------------- |
| CookieBot banner           | `essential` (default)  | Always loads                  |
| `fonts.js`                 | `essential` (default)  | Always loads                  |
| Google Analytics           | `analytics` (default)  | Loads after analytics consent |
| PostHog                    | `analytics` (default)  | Loads after analytics consent |
| Clearbit                   | `marketing` (default)  | Loads after marketing consent |
| Intercom                   | `essential` (default)  | Always loads                  |
| `ad-pixel.js`              | `marketing` (explicit) | Loads after marketing consent |
| Documentation.ai telemetry | `analytics` (explicit) | Loads after analytics consent |

## What happens after a visitor makes a choice

When a visitor accepts or rejects categories in your CMP, Documentation.ai reevaluates which integrations, scripts, and telemetry are allowed to run. Essential items keep loading, while categorized items only run when their assigned category is allowed.

If you are testing your setup, publish your changes, open the site in a fresh browser session, and use your consent banner to confirm that analytics, chat, marketing tools, and custom scripts behave the way you expect.

## Troubleshooting

**CMP banner not showing:** Add the consent banner as a [custom script](/customize/custom-scripts). Cookie consent settings only control when other scripts load. Make sure your CMP script has no `category` field, or explicitly keep it `essential`.

**Scripts loading without consent:** Check that the `category` field on the integration or script exactly matches a key in `integrations.cookies`. Category names are case-sensitive.

**Consent not detected:** Confirm that the configured `key` and `value` match what your CMP actually stores in the browser.

**Telemetry still tracking:** If you disabled telemetry, make sure `enabled` is the string `"false"`, not the boolean `false`. If telemetry should wait for consent, confirm that `integrations.telemetry.category` matches a configured cookie category.
