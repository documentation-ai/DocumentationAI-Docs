## Overview

Control who can view your documentation by configuring access control settings. You can make your documentation fully public, fully private, or a mix of both.

Access control is configured in the dashboard under **Settings > Access Control** and takes effect after you publish (or re-publish) your documentation.

<Callout kind="alert" collapsed="false">
  If you store your documentation source in your own Git repository, keep that repository private so the source content is not publicly visible. Access control settings apply only to the deployed documentation site, not to the visibility of your Git repository.
</Callout>

## Access modes

- **Public**: Anyone can access your documentation. No authentication required.

- **Private**: All pages require authentication. Choose from Password, JWT, or OAuth 2.0.

- **Partial**: A mix of public and password-protected pages. Only password authentication is available in this mode.

## Authentication methods

Documentation.AI offers three authentication methods to protect your documentation:

| Method                                         | Available in     | User-level access    | Personalization |
| ---------------------------------------------- | ---------------- | -------------------- | --------------- |
| [Password](/customize/access-control/password) | Private, Partial | No (shared password) | No              |
| [JWT](/customize/access-control/jwt)           | Private          | Yes (per-user roles) | Yes             |
| [OAuth 2.0](/customize/access-control/oauth)   | Private          | Yes (per-user roles) | Yes             |

### Password

Simple shared password protection. Best for internal docs shared with a small team.

- In **Private** mode, all pages are protected.

- In **Partial** mode, you configure which pages are public and which require the password.

[Set up password authentication ->](/customize/access-control/password)

### JWT

Integrate with your existing authentication system using JSON Web Tokens. Supports per-user access roles and personalization. Only available in **Private** mode.

[Set up JWT authentication ->](/customize/access-control/jwt)

### OAuth 2.0

Integrate with OAuth 2.0 or OpenID Connect (OIDC) providers like Okta, Auth0, or Azure AD. Supports per-user access roles and personalization. Only available in **Private** mode.

[Set up OAuth 2.0 authentication ->](/customize/access-control/oauth)

## Role-based access control

With JWT and OAuth 2.0 authentication, you can control which pages each user can access using the `pathAccessRoles` property in your `documentation.json`.

`pathAccessRoles` maps page paths to arrays of role names. Users whose token includes at least one matching role can access that path. Paths that are not listed remain visible to all authenticated users. A user with the wildcard role `*` can access every restricted path.

<Callout kind="info" collapsed="false">
  Role-based access control is only available with JWT and OAuth 2.0 authentication in Private mode. Partial mode uses password authentication only.
</Callout>

### How it works

Add a `pathAccessRoles` object at the top level of your `documentation.json`. Each key is a page path, and each value is an array of allowed roles for that page.

```json
{
  "navigation": {
    "tabs": [
      {
        "tab": "Guides",
        "groups": [
          {
            "group": "Getting Started",
            "pages": [
              { "title": "Introduction", "path": "intro" },
              { "title": "Quick Start", "path": "quick-start" }
            ]
          },
          {
            "group": "Engineering",
            "pages": [
              { "title": "Architecture", "path": "engineering/arch" },
              { "title": "Runbooks", "path": "engineering/runbooks" }
            ]
          }
        ]
      },
      {
        "tab": "Admin",
        "groups": [
          {
            "group": "Operations",
            "pages": [
              { "title": "Deploy Guide", "path": "admin/deploy" }
            ]
          }
        ]
      }
    ]
  },
  "pathAccessRoles": {
    "engineering/arch": ["engineering", "platform"],
    "engineering/runbooks": ["sre"],
    "admin/deploy": ["admin"]
  }
}
```

In this example:

- **Introduction** and **Quick Start** are visible to all authenticated users because their paths are not listed in `pathAccessRoles`.

- **Architecture** is visible to users with the `engineering` or `platform` role.

- **Runbooks** is visible only to users with the `sre` role.

- **Deploy Guide** is visible only to users with the `admin` role.

### Inheritance rules

- **Per-path matching**: Role restrictions are applied by page path, not by where the page appears in the navigation tree.

- **No navigation inheritance**: Adding a group or tab in `navigation` does not automatically apply roles to every page inside it.

- **Unlisted paths stay open**: If a page path is not present in `pathAccessRoles`, any authenticated user can access it.

- **Wildcard access**: A user whose `accessRoles` claim includes `*` can access every restricted page.

### Navigation filtering

When a user authenticates with JWT or OAuth 2.0, the sidebar, search results, and previous or next page navigation are filtered by that user's roles. Pages the user cannot access do not appear anywhere in the authenticated UI.

If a user navigates directly to a restricted URL, they see an "Access Restricted" page instead of the content.

### JWT claims format

Your JWT or OAuth provider must include the `accessRoles` field in the token claims:

```json
{
  "host": "docs.example.com",
  "firstname": "Alice",
  "company": "Acme Inc",
  "accessRoles": ["engineering", "sre"]
}
```

The field names are fixed: `host`, `firstname`, `company`, and `accessRoles`.
