# ParamField Component

The ParamField component is specifically designed for documenting API parameters, providing a structured way to describe parameter names, types, requirements, and usage details.

## Overview

Use ParamField components in API documentation to clearly describe request parameters, query parameters, path parameters, and configuration options with consistent formatting and type information.

## Properties

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `path` | `string` | Yes | The parameter name or path |
| `paramType` | `string` | Yes | The parameter data type |
| `required` | `boolean` | No | Whether the parameter is required |
| `children` | `ReactNode` | Yes | Description of the parameter |
| `defaultValue` | `string` | No | Default value if parameter is optional |
| `example` | `string` | No | Example value for the parameter |
| `className` | `string` | No | Additional CSS classes for styling |

## Usage Examples

### Basic Parameter

```mdx
<ParamField path="userId" paramType="string" required={true}>
  The unique identifier for the user account.
</ParamField>
```

### Optional Parameter with Default

```mdx
<ParamField path="limit" paramType="number" defaultValue="10">
  The maximum number of items to return in the response.
</ParamField>
```

### Parameter with Example

```mdx
<ParamField path="email" paramType="string" required={true} example="user@example.com">
  The user's email address used for authentication and notifications.
</ParamField>
```

### Complex Parameter

```mdx
<ParamField path="filters" paramType="object">
  Optional filters to apply to the search results.
  <List type="unordered">
    <li><code>category</code> (string) - Filter by category name</li>
    <li><code>dateRange</code> (object) - Date range filter with start and end dates</li>
    <li><code>tags</code> (array) - Array of tag names to filter by</li>
  </List>
</ParamField>
```

### Nested Parameters

```mdx
<ParamField path="user" paramType="object" required={true}>
  User information object containing the following fields:
</ParamField>

<ParamField path="user.name" paramType="string" required={true}>
  The user's full name.
</ParamField>

<ParamField path="user.email" paramType="string" required={true}>
  The user's email address.
</ParamField>

<ParamField path="user.age" paramType="number">
  The user's age in years.
</ParamField>
```

### Query Parameters

```mdx
<Heading level={3}>Query Parameters</Heading>

<ParamField path="page" paramType="number" defaultValue="1">
  The page number for pagination (1-based indexing).
</ParamField>

<ParamField path="per_page" paramType="number" defaultValue="20">
  Number of items per page (maximum 100).
</ParamField>

<ParamField path="sort" paramType="string" defaultValue="created_at">
  Sort field. Available options: <code>created_at</code>, <code>updated_at</code>, <code>name</code>.
</ParamField>

<ParamField path="order" paramType="string" defaultValue="desc">
  Sort order. Either <code>asc</code> or <code>desc</code>.
</ParamField>
```

### Path Parameters

```mdx
<Heading level={3}>Path Parameters</Heading>

<ParamField path="organizationId" paramType="string" required={true}>
  The unique identifier of the organization.
</ParamField>

<ParamField path="projectId" paramType="string" required={true}>
  The unique identifier of the project within the organization.
</ParamField>
```

## Parameter Types

Common parameter types you can use:

- `string` - Text values
- `number` - Numeric values
- `integer` - Whole numbers
- `boolean` - True/false values
- `array` - List of values
- `object` - Complex nested data
- `date` - Date values
- `datetime` - Date and time values
- `enum` - Predefined set of values
- `file` - File uploads

## Accessibility

- Uses semantic markup with proper labels
- Provides clear type and requirement information
- Supports keyboard navigation and focus management
- Compatible with screen readers and assistive technologies

## Best Practices

- Always provide clear, descriptive parameter descriptions
- Include examples for complex parameters
- Specify whether parameters are required or optional
- Use consistent naming conventions
- Group related parameters logically
- Document parameter constraints and validation rules
